# XiuXianBot 全面模块功能审查报告

日期：2026-03-17
范围：core/routes、core/services、core/database、adapters/telegram、web_local、web_public、tests/scripts
方式：并行多 agent 审查 + 人工复核 + 全量测试

## 测试现状
- `pytest -q`：172 项通过
- 说明：现有自动化全绿，但存在若干并发一致性与测试假阳性风险

## 关键问题清单（按严重级别）

### Critical
1. 突破流程事务一致性缺陷
- 位置：`core/services/settlement_extra.py:635,686,694,709`
- 触发：突破先扣精力，后续主事务失败（材料/并发/状态）
- 影响：失败也可能扣精力；并发下铜币可被扣成负值风险
- 建议：把体力扣减放入同一事务；铜币扣减改成 `WHERE copper >= ?` 的条件更新并检查 `rowcount`

### High
2. request_id 幂等去重竞态
- 位置：`core/database/migrations.py:355`，以及 `core/services/settlement.py:1051`、`alchemy_service.py:109`、`resource_conversion_service.py:100`
- 触发：同 request_id 并发重放/超时 takeover
- 影响：可能重复执行业务或双方都认为已抢到处理权
- 建议：stale takeover 加 CAS 条件；所有带 request_id 的写操作统一 `reserve_request()`

3. 用户唯一索引“跳过创建”导致脏数据长期存在
- 位置：`core/database/connection.py:509,524`，`core/services/account_service.py:42`
- 触发：历史数据有重复 username/telegram_id
- 影响：账号映射歧义、并发注册冲突不可控
- 建议：先去重迁移，再强制建唯一索引；注册走数据库约束主导

4. 管理端配置保存可写坏配置
- 位置：`web_local/app.py:327`
- 触发：`POST /config` 提交 `null`/非对象 JSON
- 影响：配置先写坏再报错
- 建议：先校验结构再落盘；使用临时文件+原子替换

5. 管理端删用户级联不完整
- 位置：`web_local/app.py:786,822`
- 触发：删除活跃用户
- 影响：残留孤儿数据（battle/event/economy 等表）
- 建议：补齐所有含 user_id 子表，或改外键级联/服务层统一删除

### Medium
6. 管理页用户 ID 解析 bug
- 位置：`web_local/templates/admin.html:177,319`
- 触发：搜索用户后点击“应用更改”
- 影响：提交 user_id 带中文前缀，后端查不到
- 建议：用隐藏字段或 `data-user-id`，不要从展示文本反解析

7. 公网入口暴露管理面板地址
- 位置：`web_public/app.py:39`，`web_public/templates/index.html:27,46`
- 触发：访问公开首页
- 影响：提升攻击面与信息泄露风险
- 建议：移除管理面板地址展示与复制按钮

8. Header/JSON 密码鉴权分支缺少 CSRF 约束
- 位置：`web_local/app.py:104,112`
- 触发：浏览器环境下调用敏感 POST
- 影响：比 session 分支更易被跨站滥用
- 建议：状态变更接口统一 CSRF；Header 鉴权限制到服务间调用

9. Telegram 参数错误被统一吞成“服务器错误”
- 位置：`adapters/telegram/bot.py:4792,4867,4912,4958`
- 触发：如 `/sect donate abc`、`/gacha abc`
- 影响：用户提示差、定位困难
- 建议：对 `int()` 参数做显式校验并返回可读错误

10. Smoke 测试存在假阳性风险
- 位置：`scripts/smoke_api_e2e.py:21`，`scripts/smoke_bot_layer.py:518`，`tests/test_smoke_api_e2e.py:10`
- 触发：业务回归为 4xx 或关键分支被跳过
- 影响：测试仍可能通过
- 建议：关键 happy path 强制 200+success；关键回调“缺失即失败”；增加 checkpoint 断言

## 补充说明
- 本次并行审查共使用多个 agent；其中 1 个 routes 专项 agent 被中断，已由人工复核补位。
- 目前未发现除注册接口外的明显写路由鉴权缺口（多数写路由已走 actor 校验）。

## 建议修复顺序
1. 先修 Critical/High：突破事务一致性、request_id 幂等原子化、配置写入校验、删除级联完整性
2. 再修 Medium：管理页 ID 解析、公开页暴露面、Telegram 参数校验
3. 最后加强测试：收紧 smoke 判定条件，补并发与错误分支回归用例
